## 3. Numbering / structural problems

### 3a. Dropped leading "1" in 5-digit star numbers — FIXED
On deep pages the model dropped the leading digit of 5-digit star numbers (e.g. wrote
`1751` for `11751`). Because the catalog is strictly increasing past 10934, any star
number `< 10000` on pages 215+ is unambiguously this error → fix is `+10000`.

| Page | Rows fixed |
|------|-----------|
| 231 | 10 |
| 236 | 12 |
| 239 | 12 |
| 241 | 59 (entire page) |
| 245 | 56 (entire page) |
| 372 | 13 (rows 9156–9168 → 19156–19168) |
| 408 | 12 (rows 1171–1182 → 21171–21182) |

On pages 372 and 408 the model dropped the leading digit only for the first chunk of
rows and then self-corrected partway down the page (so the tail already had the full
number). The page endpoints (confirmed by the neighbouring pages' seams: 372 = 19156–19212,
408 = 21171–21230) made the true range unambiguous, so the early rows were lifted by
+10000 (372) / +20000 (408); both pages are now perfectly consecutive internally.
**The numbering validator does NOT catch this — re-scan every future deep batch for
star numbers `< 10000` and for non-monotonic page seams.**

### 3b. Layout guard false-positive on dark pages — SCRIPT FIXED
Pages **43** and **61** were skipped with `suspicious Mag column width`. Cause: these are
darker scans whose vertical rules print bold/bleed wide (44–53 px vs the normal 7–8 px),
squeezing the white Mag *gap* to 26–31 px and tripping the `35 ≤ gap` floor — even though
the actual crop captured every magnitude. Fix in `extract_gc_catalog.py` (~line 249):
the guard now validates the **crop width** (`40 ≤ width(crops[2]) ≤ 200`), i.e. the region
actually sent to the model, not the white gap. Both pages re-extracted cleanly afterward.

### 5c. Designation-system notes (for verifying references)
- **St.** = Stone → `results/cross/cross_stone_ppm.csv` keys `"St N"`.
- **F.** = Flamsteed → key `"<num> <Const>"` in `cross_ua_ppm` (no `G`). Gaps are real
  omissions (faint stars below the UA limit, e.g. 23 Mon = GC 9117).
- **UA** → key `"<num>G <Const>"`; component letters a/b denote doubles (point to comp. a);
  fractional UA numbers (e.g. "272½") are anonymous ua.txt entries, verifiable via HD/SAO.
- **ZC. / WB.** → key `"<cat> <H>h <num>"` with H = RA hour; allow H−1 if RA min < 5,
  H+1 if min > 55 (epoch-1825 RA can place a star in the adjacent hour zone, e.g.
  GC 5825 = WB 4h 1355, GC 7250 = WB 5h 1527).
- **Y.** = Yarnall/USNO → convert via `yarnall/yarnall_usno.txt`. Verified pairs:
  Y 1291=U 1331, Y 1313=U 1355, Y 1315=U 1358, Y 1316=U 1359, Y 1360=U 1406.
- Parenthesized references (e.g. `L. (589)`) are uncertain in the book and are skipped by
  `readGCScanned` for cross-checking.

### 5d. Reference validation of magnitude-flagged rows
Rationale: when Gemini **misread the magnitude** of a row (a §2a CORRECTED entry), it may
have struggled with that whole row, so its **reference** is also suspect → verified against
the print. Rows where the print simply differs from gc.txt (§2b KEPT) need no such check —
the cause is gc.txt, not Gemini. Reference reads must use the `compose()`-aligned image
(num+mag+ref); a raw crop spanning far-apart columns misaligns because the print rows lean
up to half a row across the page (the row anchor is the **magnitude**, not the number).

Result for the §2a CORRECTED rows: the reference is **correct** for essentially all of them
(the mag glyph and the reference are independent crops, so a fraction misread rarely touches
the reference). Two genuine reference defects were found on the same rows whose magnitude was
wrong, plus the drop-page cases:

## 6. Pipeline facts (for whoever resumes)
- Model `models/gemini-3.1-flash-lite`; free tier **15 RPM / 500 req/day**; `CALL_DELAY=5s`.
  Per-day quota aborts cleanly with HTTP 429 `RESOURCE_EXHAUSTED`.
- API key is read from `../../gapikey.txt` (outside the repo) or `$GOOGLE_API_KEY`.
- Render: `pdftoppm` 300 DPI, cached in `/tmp/gc_pages`.
- Validation: numbers consecutive + chain anchor (previous page's CSV) + magnitudes equal
  `cat/gc.txt`; 3 runs max, references need 2 agreeing runs.
- **gc.txt magnitude format:** cols 8–10. If the stripped field is 3 chars it is mag×10
  ("065"→6.5, "082"→8.2); if 2 chars it is a whole magnitude ("09"→9.0, "11"→11.0).
  A naive always-÷10 parser is WRONG for whole magnitudes — use `load_gc_magnitudes()`.
